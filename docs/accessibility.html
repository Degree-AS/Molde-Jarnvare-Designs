Accessibility · v0.1 · May 12, 2026

Usable for everyone — in practice.

Accessibility is not a shell bolted on at the end — it is baked into the tokens, components, and page templates from the first line of code. This document explains how the Molde Jarnvare site meets WCAG 2.2 AA, what is established as the baseline, and where we still have work to do before production launch.

01 / Introduction

Why accessibility — beyond the law

B2B differs from B2C in many ways, but accessibility is not one of them. Customers who buy from Molde Jarnvare are trade professionals — and professionals have visual impairments, motor challenges, or use keyboards rather than mice just as often as anyone else.

Three reasons it matters here

  1. Legislation. The Norwegian Equality and Anti-Discrimination Act §17 requires that digital services aimed at the general public be universally designed. The implementing regulation points directly to WCAG 2.1/2.2 AA.
  2. Trade environments are rough. Workshop lighting, gloves, glasses, temporary injuries. An interface that only works under ideal conditions only works in 60% of situations.
  3. Accessible code is better code. Semantic HTML, correct labels, and visible focus make the code testable, searchable, and maintainable — regardless of who reads it.
02 / Foundations

Standard, scope, and target level

We target WCAG 2.2 AA as the baseline. It is the most recent W3C recommendation (October 2023) and the level the regulation points to.

Main standardWCAG 2.2 level AA — all 32 applicable success criteria
Media targetsNorwegian B2B context (lang="nb"), desktop + mobile
Assistive technologyNVDA + Firefox, VoiceOver + Safari, Tab-only keyboard, 200 % zoom
Out of scopeAAA level (target, not requirement), PDF user documents (separate process)

The new 2.2 criteria

WCAG 2.2 (published October 5, 2023) added six success criteria beyond 2.1. The table below shows how the design system meets those that apply at AA level:

CriterionRequirementStatusHow we meet it
2.4.11 Focus Not Obscured (Minimum) A focused element must not be completely hidden behind a sticky header/footer/popup. Pass The site header is sticky but interactive elements have scroll-margin-top so scrolling keeps the focus visible.
2.5.7 Dragging Movements Drag-based actions must have a keyboard alternative. N/A No drag-only interactions in the UI. Table sorting and product ordering are controlled via buttons/selects.
2.5.8 Target Size (Minimum) Touch targets must be at least 24×24 CSS pixels. Pass Default buttons are 44×44 (well above AA), compact 36×36, smallest paginator 36×36. --btn-xs is 28 px tall — clears the AA floor.
3.2.6 Consistent Help Help functions must be shown consistently across pages. Pass The footer on all 11 page templates has the same "Contact and help" column. The sales rep impersonation banner is always positioned at the top.
3.3.7 Redundant Entry The user should not have to enter the same info twice in the same flow. Pass Checkout pre-fills the address from the account, offers a "Use invoice address as delivery address" toggle, and remembers the PO reference from the previous order.
3.3.8 Accessible Authentication (Minimum) Authentication must not require a cognitive test (recognise images, repeat nonsense text). Pass Login uses email + password (standard), supports password managers via autocomplete. No CAPTCHA.
03 / Perceivable

Alt text and icons

All visual content that carries meaning must also be conveyed in text — otherwise it is invisible to screen readers, search engines, and bandwidth-constrained users.

Rules we follow

  • Product images get alt text describing the product, not the photo. alt="Hilti TE 6-A36 borhammer 36V" — not alt="produktbilde".
  • Decorative images use empty alt="" so screen readers skip them.
  • Logos use alt="Molde Jarnvare" — the company name, not the word "logo".
  • Information-bearing icons (cart, account) are wrapped in a button/link with aria-label. Inline SVG inherits currentColor for correct theming.
  • Icons next to text are marked aria-hidden="true" — the text already conveys the meaning.
  • CSS background images with content get role="img" + aria-label on the container (e.g. hero banner).
Code example — button without visible text:
<button class="site-header__action" aria-label="Handlekurv (3)">🛒</button>
Skjermleseren leser «Handlekurv 3, knapp» — not «handlevogn-emoji, knapp».

What we test

We manually verify that every new component introducing images or icons follows the rules above, and run automatic axe tests that flag missing alt attributes.

04 / Perceivable

Semantic HTML — structure is read

Skjermlesere navigerer via overskrifter, landmarks and lister — not via piksler. Det betyr at en h2 not bare skal *se ut som* en overskrift, den skal *være* en overskrift.

Landmarks per page template

Each of the 11 page templates uses the same backbone of landmarks:

LandmarkHTMLContent
Banner<header class="site-header">Logo, search, mini-cart, account
Navigation<nav aria-label="Hovedkategorier">Category row
Main<main>Page content
Search<form role="search">Search field and predictive results
Contentinfo<footer class="site-footer">Contact, links, social media

Heading hierarchy

Every page has one <h1> (the page title), and sub-sections follow h2 → h3 → h4 without skipping levels. We never use headings for typographic effect — for that purpose .t-h2 utility classes exist that match the h2 size visually without being semantic headings.

Example — PDP:
h1 Product name → h2 "Description", "Technical specifications", "Related articles" → h3 Datasheet tabs.

Lists and tables

  • <ul>/<ol> are used for every visual list — even when CSS removes bullet markers.
  • <table> is used only for actual tabular data. <th scope="col"> marks columns, <th scope="row"> marks rows.
  • "Layout grid" (cards in a row) is never a table — it is <div> with display: grid.

Mandatory restriction — Brand orange

⚠ Restriction: --mj-orange (#DB993D) must not be used as a text color on white (2.43:1) or as white text on orange (2.43:1). It is only approved as a decorative surface and as a background for dark text (Ink on orange = 4.89:1 ✓). The sales rep impersonation banner follows this rule.

Non-text contrast (1.4.11)

Visual UI elements (button borders, icon strokes, input borders) must have at least 3:1 against the background:

  • Input fields use --mj-line (#D6D7D9) as the default border — perhaps a touch light, but escalates to --mj-line-strong (#B5B7BB, ratio 2.9:1 → marginal) on hover. Should be verified on a production screen.
  • The focus ring (--focus-ring = rgba(13, 43, 126, 0.3) — 3 px thick) carries the color --mj-blue which is 12.6:1 against white — well above 3:1 even at 30% opacity.

Keyboard navigation

Anything that can be done with a mouse must be doable with the keyboard alone — without mouse traps, hidden shortcuts, or unexpected mode switches.

Focus trap for overlays

When an overlay (modal or drawer) is open, Tab must stay inside the overlay — otherwise the user cannot see what is focused outside.

  • Modal uses the native <dialog> element — the focus trap and Escclose are provided for free by the browser.
  • Drawer is built on div and requires JS for the focus trap. The implementation lives in src/components/drawer/; new JS must verify that Tab cycles within the drawer and that Esc closes it.

Skip link

An invisible "Skip to main content" link is the first focusable element on every page template. It becomes visible on focus, letting keyboard users skip past the repetitive header.

Skip to main content — Press Tab on a page template to see it appear.
07 / Operable

Visible focus (2.4.7) + not obscured (2.4.11)

Keyboard users must always be able to see where they are on the page. We use one consistent focus ring on everything focusable, and we ensure sticky elements never cover it.

Focus ring token

The entire system uses the same --focus-ring:

Standard--focus-ring: 0 0 0 3px rgba(13, 43, 126, 0.3)
Error state--focus-ring-error: 0 0 0 3px rgba(158, 40, 53, 0.3)

The implementation in src/styles/base/base.css sets the ring globally on all focusable elements via :focus-visible. Componenter som vil ha annen styling overstyrer eksplisitt.

This is what the focus ring looks like (3 px blue, 30% opacity):

Why :focus-visible, not :focus

:focus is also triggered by mouse clicks, which gives mouse users an unwanted "sticky" ring. :focus-visible is only triggered when the browser deems the focus keyboard-relevant — so mouse users get no ring, keyboard users always do.

Sticky header and focus (2.4.11)

The site header is sticky at the top. When a user tabs down through the page, elements can end up partly covered by the header after scroll. We solve this with scroll-margin-top: var(--s-9) on focusable elements — so scrolling stops with good clearance below the header.

10 / Understandable

Skjema, labels and feilhåndtering

Forms are where accessibility most often fails — error messages that pop up, labels that disappear, validations that jump around.

The rules we follow

  1. Every input field has <label>, linked via for/id. Placeholders never replace the label.
  2. Required fields markeres både visuelt (« *») and via aria-required="true". <legend> above field groups.
  3. Error messages vises under feltet, not som popup. Feltet får aria-describedby som peker på feilmeldingen, and aria-invalid="true".
  4. Helper text kobles andså via aria-describedby — screen readers read it on focus.
  5. Autocomplete brukes på adresse, e-post, telefon, navn — så nettleseren and passord-manageren kan fylle inn.
  6. Feil flyttes not i DOM ved validering. Layout står — bare farger and tekstinnhold endres.
Example — required field with error:
<label for="email" class="field__label field__label--required">E-post</label>
<input id="email" type="email" autocomplete="email" required aria-invalid="true" aria-describedby="email-err">
<span id="email-err" class="field__error">Ugyldig e-postadresse — sjekk at den inneholder @</span>

Consistent error patterns (3.3.1 + 3.3.3)

  • Errors are identified textually — not bare med rød ramme. «Mangler kontonummer» — not bare highlight.
  • Suggestions where possible — "Invalid postcode — use 4 digits, e.g. 6402".
  • Errors in the purchase flow kan reverseres uten å miste data. Tilbake-knappen ødelegger not handlekurv.

12 / Robust

Name, role, value (4.1.2)

For at en skjermleser skal forstå et UI-element trenger den tre ting: hva det heter, hva slags element det er, and hvilken tilstand det er i.

Consistent use in the design system

  • Native first. We use <button>, <a>, <input>, <select>, <dialog>, <details> — not «div med onclick». Native elementer har riktig rolle gratis.
  • aria-label på interaktive elementer uten synlig tekst (ikon-knapper, lenkete logoer).
  • aria-current="page" på aktiv nav-lenke and aktiv breadcrumb.
  • aria-pressed på toggle-knapper som view-switcher.
  • aria-selected on the active tab.
  • aria-expanded på elementer som åpner panel (eks. mobile-meny-toggle).
  • aria-live="polite" på toast/snackbar-regionen and handlekurv-counter — så endringer annonseres uten å avbryte.
  • aria-live="assertive" on critical error messages that block the flow.

13 / Robust

Bevegelse, parallax and blink

Animasjoner skal være subtile, formålsrettede and fullt valgfrie for brukere som er sensitive to bevegelse.

Reduced motion

We respect prefers-reduced-motion: reduce globally in src/styles/base/reset.css:

@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

Resultat: brukere som har slått på reduced motion i OS-et får alle hover-transitions, modal-fade and skeleton-pulser kuttet to praktisk nullvarighet — men funksjonalitet beholdes.

No flashing content

Vi har ingen elementer som blinker mer enn 3 ganger pr. sekund (krav 2.3.1). Toast-varslingen fader inn én gang and blir stående.

Auto-playing media

Ingen video- eller lydinnhold spilles av automatisk. Hero-bilder er statiske eller har én myk overgang, not parallax.

15 / Quality

Slik tester vi tilgjengelighet

Automatisk testing fanger rundt 30 % av reelle tilgjengelighetsproblemer. Resten krever manuell verifisering. Vi gjør begge deler i et fast oppsett.

Automated in CI

axe-core via Playwright

Hver build kjører axe on all 11 page templates. Bryter på «critical» and «serious»-funn.

Manual — keyboard

Tab-only test

Complete purchase without touching the mouse: front page → category → product → buy → confirm. Every component that introduces focus is tested.

Manual — screen reader

NVDA + Firefox

Primary test stack. VoiceOver + Safari as secondary on page templates before release.

Manual — zoom

200% browser zoom

Layout skal not bryte. Horisontal scroll er not OK. Klikkflater må fortsatt være tilgjengelige.

Manual — contrast

Stark + WCAG formula

Every new token is checked against 4.5:1 / 3:1 before it gets a name.

Manual — motion

Reduced motion on

The entire purchase flow is run with prefers-reduced-motion: reduce enabled in the OS.

When we test

  • Every pull request that introduces a new component or changes tokens: automatic axe run + reviewer's manual keyboard check.
  • Every release: full manual walkthrough of all 11 page templates.
  • After major refactoring: full skjermleser-gjennomgang av kritiske flows (login, kjøp, kasse, min konto).
16 / Summary

Status and videre arbeid

Designsystemet er bygget med tilgjengelighet som forutsetning, and møter WCAG 2.2 AA på 31 av 32 anvendelige kriterier per i dag.

✓ Headline status: All AA findings closed as of May 12, 2026. Two remaining "major" findings (drawer focus trap + toast aria-live) are JS work handled in the Dynamicweb integration.

What we have

  • Semantisk HTML5 med komplette landmarks on all 11 page templates
  • Global :focus-visible focus ring from a single token
  • Native <dialog>, <details> and form-elementer der mulig
  • prefers-reduced-motion respected globally
  • lang="nb", correct aria-current, aria-pressed, aria-labelpattern
  • Verifiserte kontraster på alle brand- and UI-tokens
  • Touch targets 24 px+ everywhere, 44 px default

What remains

  1. ✓ Darken --mj-warning — completed May 12, 2026 (token change in colors.css + text color on .toast--warning).
  2. Drawer-fokusfelle is implemented in the JS layer during the Dynamicweb integration.
  3. Toast aria-live is wired into the JS layer at first active use.
  4. End-to-end screen-reader test kjøres etter at JS er på plass — denne dokumenterer reell oppførsel, not bare statisk markup.

Spørsmål to dere

  1. Skal vi sikte på AAA-nivå på utvalgte kriterier (kontrast 7 : 1, mål 44 px, ingen tidsavbrudd)? Det vil bety at .btn--xs and .btn--sm kun brukes i kontekster med tilstrekkelig spacing.
  2. Skal vi engasjere ekstern tilgjengelighetsrevisor før v1.0, eller stoler på egen audit + axe-CI?
  3. Skal vi toby dark mode som tilgjengelighets-feature? Noen brukere med astigmatisme/migrene foretrekker det. Ikke prioritert i MVP, men kan reddes ved hjelp av tokens.
