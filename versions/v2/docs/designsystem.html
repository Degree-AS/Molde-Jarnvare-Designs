01 / Introduction

Why a design system?

When several people are going to build content, pages, and features over time — without the result feeling like a patchwork — they need a shared language. The design system is that language.

What the system gives us

01

Consistency

Same colors, fonts, spacing, and components across every page — no matter who builds them.

02

Speed

New pages are composed from ready-made building blocks. Time is saved each time.

03

Quality

Accessibility, ytelse and mobil-tilpasning er løst én gang per komponent — not på hver side.

04

Scalability

Change one color or one spacing value, and every page updates automatically.

Relationship with the 2025 brand book

The brand book owns the brand expression: lando, brand colors, slandan, tone, typandraphy in Microsoft surfaces, and image style. This design system operationalises the brand for web and B2B e-commerce — where we need additional rules for things the brand book does not cover: UI-functional status colors, a web-specific typandraphy substitute, component-specific rules, and accessibility requirements (WCAG 2.2 AA).

When this design system deviates from the brand book, the brand book takes precedence. Any such deviations are flagged and clarified with you.

02 / Foundations

Six principles behind every choice

These guide everything from color choices to component names.

01

Trade first

Technical data, stock status, and article numbers must always be clear and immediately available.

02

Partner, not supplier

The tone is "on the team with" — not a distanced sales organisation. We speak as a colleague who knows the trade.

03

Local roots

90 % of revenue comes from Møre and Romsdal. The design must feel like a place, not a generic B2B portal.

04

Breadth and depth

33,000 articles. Hierarchy, filtering, and search are core functionality, not decoration.

05

The sales rep is on board

The UI must work both as a shop for customers and as a tool for sales reps — including "shop as" mode.

06

Structured content

Content er typete blokker, not fri-tekst. Det gjør det vedlikeholdbart, søkbart and AI-kompatibelt.

Slandan

Slandanet skal not brytes opp or omformuleres.

Address

  • Use "du" (informal you), never the formal "De" / "Dem".
  • B2B context — assume a trade-skilled reader. Do not explain what a hex bolt is.
  • Address the person, not the company: "You can request a quote" — not "The company can request a quote".
  • We say "we" for Molde Jarnvare, never refer to "Molde Jarnvare" in the third person.

Usage rules

✓ Correct
  • Keep the protection area around the lando (half the letter height)
  • Use negative on Reflex blue (contrast 12.4:1 ✓)
  • Use positiv på hvit or lys grå
  • Minimum 200 px width for full lockup; 32 px for the MJ monandram alone
✗ Never

  • Change colors beyond the approved variants

  • Roter or skjev landoen

  • Use som mønster or dekorelement

06 / Visual language

Typandraphy

We use Source Sans 3 on the web — a humanist sans-serif that visually and tonally substitutes for the brand book's Aptos. Aptos is still used inside the Microsoft ecosystem (Word, PowerPoint, email).

Why a substitute? Aptos er Microsofts standardfont and er not lisensiert for @font-face på offentlige nettsider. Source Sans 3 er gratis (SIL OFL), designet for skjerm, and har tilsvarende åpne, vennlige bokstavformer.

16 / Quality

Accessibility — WCAG 2.2 AA

Accessibility er not noe vi legger på til slutt. Det er en baseline-forutsetning som er bygd inn i hver komponent — and documented i et eget audit.

✓ Status per May 12, 2026: All applicable AA success criteria met. The single critical audit finding (warning-token contrast at 2.86:1) is closed — --mj-warning darkened to #7A580E, gir nå 5,68 : 1 på status-pill and 6,50 : 1 på badge/stock/toast.

What we do

  • Contrast: Alle tekst-kombinasjoner møter 4,5 : 1 (brødtekst) or 3 : 1 (stor tekst). Brand-orange er flagget som «kun dekorativ» (2,4 : 1 mot hvit).
  • Keyboard navigation: All interactive elements have a visible focus indicator (--focus-ring). Tab order follows the DOM.
  • Touch targets: Standard knapper and inputs er 44 × 44 px. .btn--sm (36 px) and .btn--xs (28 px) clear WCAG 2.2 AA's 24 px floor.
  • Screen readers: Form fields har <label>, icon buttons have aria-label, dialander bruker native <dialand> which announces the correct role. The variant matrix has a specific aria-label per cart button ("Add M8 × 30 mm A4 stainless to cart, 100 pcs").
  • Motion: prefers-reduced-motion is respected globally in base/reset.css.
  • Structured markup: Headings follow the semantic hierarchy (one h1 per page, h2 → h3 without skipping levels). Visual styling is separated from HTML tags via utility classes.
  • Sticky-header and fokus: scroll-margin-top sørger for at fokuserte elementer not skjules bak header (WCAG 2.2 2.4.11).

What we test

  • Manual keyboard test of every component
  • Screen reader (NVDA/VoiceOver) on page templates
  • Automatisk lint via axe-core or tilsvarende i CI
  • Contrast verification for new color choices (WCAG formula + visual check)
  • 200 % browser-zoom — layout skal not bryte, ingen horisontal scroll
  • Reduced-motion flow through the entire purchase journey

Remaining work

To «major»-funn håndteres når komponentene kobles til JS-laget i Dynamicweb-integrasjonen — kan not løses i CSS alene:

  • Drawer focus trap: Modal uses native <dialand> (gratis fokusfelle), men drawer er custom and needs JS-implementasjon for Tab-sirkulering and Escclosing.
  • Toast aria-live: .toast-region needs role="status" aria-live="polite" so that screen readers announce new toast messages.
Full audit report: See the dedicated accessibility document accessibility.html — it contains the full per-criterion walkthrough, all measured contrasts, the keyboard shortcut matrix per component, and the formal audit trail.
17 / Quality

Responsive design — mobile, tablet, desktop

Our customers shop both from the warehouse floor on a phone and from a planning desk on a 1440 px monitor. Every page template must work end-to-end on both — no horizontal scroll, no hidden actions, no degraded reading experience. The same words and actions are available everywhere; only the layout differs.

Philosophy

  • Mobile-first. Base CSS describes the mobile layout. Larger screens add behavior via @media (min-width: ...), never the other way around.
  • Progressive enhancement. Tablet adds breathing room; desktop adds a second column. The core flow never changes.
  • Touch first on every viewport. Tap targets are at least 40 × 40 px (44 × 44 px preferred) regardless of screen size. This keeps the system usable with a glove on a workbench, not only on a phone.

Canonical breakpoints

The token file (src/styles/tokens/layout.css) defines six breakpoints — --bp-xs through --bp-2xl — but only three are used as layout cutoffs in practice. The other three are reserved for micro-adjustments inside a single component.
